# 用户隔离鉴权机制修复计划

## 摘要

用户报告"新创建的普通用户能看到其他用户的使用数据"，判定为用户隔离 BUG。

经 Phase 1 探索，**后端 list/get 端点其实已按 `user_id` 行级过滤（健康）**，用户感知到的"看到他人数据"主要来自**前端**：登出/登录切换时未清理 workshop-store / canvas-store / pipeline-store 的 sessionStorage 与 react-query 缓存，导致新用户 hydrate 后继承前一个用户的工坊状态、`sessionId`、画布 `projectId`、流水线 `finalResult`。后端还有一个低危防御深度漏洞：`conversation_agent` 的 chat 端点不预校验请求体 `conversation_id` 归属。

本计划同时修复前端状态隔离（主因）与后端防御深度（次因），完整解决用户感知现象。

## 假设与决策

1. **修复范围**：前端状态隔离 + 后端加固（完整方案）
2. **`/outputs` 静态文件公开挂载**：暂不处理（文件名含 uuid 难猜测，且前端展示依赖此 URL，改动成本高，超出本次用户隔离主问题范围）
3. **`get_current_admin` 死代码**：暂不处理（与本次 BUG 无关，admin 角色功能属后续需求）
4. **CORS `allow_origins=["*"]` + `allow_credentials=True`**：暂不处理（配置隐患但非本次隔离 BUG 主因）
5. **循环依赖评估**：workshop-store / canvas-store / pipeline-store 均**不** import auth-store，因此 auth-store 单向 import 它们调用 `reset()` 不构成循环。react-query 的 `queryClient` 需提升为 module-level 单例。
6. **各 store 已有 `reset()` 方法**（探索确认）：workshop-store.ts:441-449、pipeline-store.ts:146-153、canvas-store.ts:249-266，均会清除对应 sessionStorage key，可直接复用。

## 修复方案

### 决策 1：新增 `lib/session-cleanup.ts` 集中清理工具

**为什么独立文件**：logout 逻辑需协调多个 store + react-query + URL，分散在 auth-store 会引入对 queryClient 的依赖（auth-store 目前不依赖 react-query）。独立工具函数被 auth-store、use-logout hook、401 处理共用，避免重复。

**文件**：`frontend/src/lib/session-cleanup.ts`（新建）

```ts
import { queryClient } from './query-client';
import { useWorkshopStore } from '@/stores/workshop-store';
import { usePipelineStore } from '@/stores/pipeline-store';
import { useCanvasStore } from '@/stores/canvas-store';

/**
 * 清理所有用户相关的本地状态：zustand store 内存 + sessionStorage + react-query 缓存。
 * 用于登出、登录切换、401 自动登出场景。
 * 注意：不清 auth-store 的 token/user（由调用方决定时机）。
 */
export function clearAllUserData(): void {
  // 1. 重置各业务 store（reset 内部会清对应 sessionStorage key）
  useWorkshopStore.getState().reset();
  usePipelineStore.getState().reset();
  useCanvasStore.getState().reset();
  // 2. 清空 react-query 所有缓存（列表/详情/查询）
  queryClient.clear();
}
```

### 决策 3：修改 `stores/auth-store.ts`

**改动点**：
- `logout()` 在清 token/user 之前调 `clearAllUserData()`（lazy import 避免循环依赖风险）
- `setAuth()` 在 set 之前调 `clearAllUserData()`（防止 A→B 同 tab 切换继承内存状态），之后 invalidate 列表 query

```ts
// auth-store.ts 顶部不 import session-cleanup（避免循环），在函数体内 lazy import
logout: () => {
  // lazy import 避免循环依赖
  import('../lib/session-cleanup').then(({ clearAllUserData }) => {
    clearAllUserData();
    try {
      localStorage.removeItem(STORAGE_KEYS.token);
      localStorage.removeItem(STORAGE_KEYS.user);
    } catch { /* ignore */ }
    set({ token: null, user: null, isAuthenticated: false });
  });
},
```

**注意**：lazy import 是异步的，logout 变成异步语义。如果 Header.tsx 期望 logout 同步完成（如调用后立即 navigate），需调整。

**更稳妥方案**：直接顶部 import（探索确认 workshop-store/canvas-store/pipeline-store 不 import auth-store，session-cleanup 只 import 这三个 store + queryClient，不 import auth-store，无循环）：

```ts
import { clearAllUserData } from '../lib/session-cleanup';

logout: () => {
  clearAllUserData();
  try {
    localStorage.removeItem(STORAGE_KEYS.token);
    localStorage.removeItem(STORAGE_KEYS.user);
  } catch { /* ignore */ }
  set({ token: null, user: null, isAuthenticated: false });
},

setAuth: (token, user) => {
  clearAllUserData();  // 登录前清旧用户状态
  try {
    localStorage.setItem(STORAGE_KEYS.token, token);
    localStorage.setItem(STORAGE_KEYS.user, JSON.stringify(user));
  } catch { /* ignore */ }
  set({ token, user, isAuthenticated: true });
  // 登录后触发列表重新拉取
  queryClient.invalidateQueries({ queryKey: ['workshop'] });
  queryClient.invalidateQueries({ queryKey: ['conversations'] });
  queryClient.invalidateQueries({ queryKey: ['canvas'] });
},
```

采用直接 import 方案（更简单、同步）。

### 决策 4：新增 `hooks/use-logout.ts` 协调登出 + 路由跳转

**为什么需要**：auth-store.logout 不应负责路由跳转（store 不应感知路由）。需要一个 hook 在 logout 后 `navigate('/create')` 清除 URL 参数 `?sessionId`/`?conversationId`。

**文件**：`frontend/src/hooks/use-logout.ts`（新建）

```ts
import { useCallback } from 'react';
import { useNavigate } from 'react-router-dom';
import { useAuthStore } from '@/stores/auth-store';

export function useLogout() {
  const navigate = useNavigate();
  return useCallback(() => {
    useAuthStore.getState().logout();  // 内部已调 clearAllUserData
    navigate('/create', { replace: true });  // 清 URL 参数
  }, [navigate]);
}
```

## 风险与回滚

- **风险 1**：`clearAllUserData()` 在 `setAuth()` 中调用可能导致登录瞬间页面闪烁（状态先清再拉）。
  - 缓解：setAuth 后立即 invalidate queries 触发重新拉取，react-query 会用新 token 拉新数据。
- **风险 2**：lazy import vs 直接 import 循环依赖。
  - 缓解：探索已确认 workshop/canvas/pipeline store 不 import auth-store，session-cleanup 只依赖这三个 store + queryClient，无循环。直接 import 方案安全。
- **风险 3**：use-conversation 监听 isAuthenticated 可能与现有 useEffect 冲突。
  - 缓解：仅监听 false 变化调 reset，不干扰 true 时的正常流程。
- **回滚**：所有改动集中在 3 新建 + 5 修改 + 1 测试文件，git revert 即可完整回滚。
