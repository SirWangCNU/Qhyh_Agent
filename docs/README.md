## 需求沉淀规则

1. **已确认的需求及时写入 PRD**
   - 当讨论功能实现、产品需求、业务规则或交互细节时，若某些需求已经明确，请及时整理并写入 [`docs/prd.md`](./prd.md)。
   - 写入 PRD 前，需基于**已确认的信息**，不要把未经确认的猜测当成最终需求。

2. **不完整需求标记为待确认**
   - 如果需求还不完整，可以先在 [`docs/prd.md`](./prd.md) 的「待确认问题 / 开放问题」章节记录为“待确认问题”或“开放问题”。

3. **详细技术方案保留在 `.trae/documents/`**
   - 已实现的功能方案、改造计划等过程文档保留在 [`.trae/documents/`](../.trae/documents/)，PRD 中通过链接引用，避免重复。

4. **API 变更同步到 `docs/api/`**
   - 新增、修改或废弃接口时，同步更新 [`docs/api/endpoints.md`](./api/endpoints.md)。

5. **图片资料归入 `docs/images/`**
   - 流程图、架构图、UI 截图、参考图等按功能模块分文件夹存放，命名规范见 [`docs/images/README.md`](./images/README.md)。
