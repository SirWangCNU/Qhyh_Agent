# 历史记录持久化与加载时序优化计划

## 一、现状与根因

### 1.1 两类历史记录的"刷新才显示"问题

| 类型 | 数据源 | 根因 |
|---|---|---|
| 对话记录 Plan | localStorage `qinghe_plans` | `use-plans.ts` 用组件级 `useState`，`SidebarNewPlan` 与 `SidebarPlanList` 各持独立 state；`storage` 事件仅跨 tab 触发，同 tab 写入不感知 |
| 工坊记录 WorkshopSession | 后端 SQLite + react-query | `useWorkshopSessions` 挂载即查询，早于 `App.useEffect` 的 `hydrate()`；token 未恢复 → 401 → `apiFetch` 调 `logout()` 清空 localStorage token；登录后无 auth 变化监听 |

### 1.2 附带问题

- `conversation_agent` 三个端点完全无持久化，对话历史不落库。
- `use-plans.ts` 的 `updatePlan` / `removePlan` / `getPlan` 在全项目零调用方。
- `use-chat-pipeline.ts` 的 `messages` 从不写回 plan，localStorage 里的 `plan.messages` 永远为 `[]`。
- `apiFetch` 的 401 处理在"从未登录/token 未恢复"时也调用 `logout()`，破坏性清除 localStorage。

## 二、设计决策

1. **对话记录迁移到后端 SQLite**：与 `workshop_sessions` 风格一致，支持跨设备同步、分页查询、后续扩展。
2. **采用关系型两表**：`conversations` + `conversation_messages`。对话消息天然关系型，支持按消息分页、未来全文检索；避免单表 JSON 膨胀。
3. **工坊记录不新建表**，仅修复加载时序与 auth 响应。
4. **前端 react-query 统一管理**对话记录，与工坊记录一致；废弃 `use-plans.ts` 的 localStorage 方案。
5. **auth hydrate 提前到模块顶层**，消除 query 早于 hydrate 的竞态。

### 3.2 表结构（关系型两表）

**`conversations` 表**（迁移 `005_create_conversations.py`）：

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | String(36) | PK | UUID hex |
| user_id | Integer | FK(users.id), NOT NULL, INDEX | 所有者 |
| title | String(200) | NOT NULL | 默认取首条用户消息前 30 字 |
| summary | String(500) | NULL | 可选摘要（预留） |
| iterations | Integer | NOT NULL DEFAULT 0 | ReAct 总迭代数（冗余，便于列表展示） |
| message_count | Integer | NOT NULL DEFAULT 0 | 消息数（冗余，避免 count 查询） |
| created_at | DateTime | NOT NULL, server_default=now() | |
| updated_at | DateTime | NOT NULL, server_default=now(), onupdate=now() | |

索引：`(user_id, updated_at DESC)` 复合索引，加速列表查询。

**`conversation_messages` 表**：

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | String(36) | PK | UUID hex |
| conversation_id | String(36) | FK(conversations.id) ON DELETE CASCADE, NOT NULL, INDEX | |
| seq | Integer | NOT NULL | 消息序号（同会话内递增，用于排序） |
| role | String(16) | NOT NULL | user / assistant |
| type | String(16) | NOT NULL DEFAULT 'text' | text / react |
| content | Text | NOT NULL DEFAULT '' | 消息正文 |
| meta_json | Text | NULL | ReAct 事件等结构化元数据 |
| created_at | DateTime | NOT NULL, server_default=now() | |

索引：`(conversation_id, seq)` 复合索引，加速分页。

### 3.3 API 端点

`qinghe-video/src/conversation_sessions/router.py`，全部 `Depends(get_current_user)`：

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/conversation-sessions` | 创建会话，body: `{title?, first_message?}`；返回完整会话 |
| GET | `/api/conversation-sessions` | 列表，分页 `?page=1&page_size=20`；返回 `ConversationSummaryDTO[]`（不含 messages） |
| GET | `/api/conversation-sessions/{id}` | 详情，含全部 messages（按 seq 升序） |
| PUT | `/api/conversation-sessions/{id}` | 更新标题 |
| POST | `/api/conversation-sessions/{id}/messages` | 追加单条消息（user 或 assistant）；自动更新 `message_count`、`updated_at` |
| DELETE | `/api/conversation-sessions/{id}` | 删除会话（CASCADE 删消息） |

**Pydantic 模型**（`schemas.py`）：
- `ConversationCreateRequest`、`ConversationSummaryDTO`、`ConversationDetailDTO`、`MessageDTO`、`MessageCreateRequest`

### 3.4 conversation_agent 集成

修改 `src/conversation_agent/router.py` 的 `POST /api/conversation/chat`：
- 接收可选 `conversation_id` 参数。
- 若提供：将本轮 user 消息和 assistant 最终 answer（含 ReAct events）写入 `conversation_messages`。
- 若未提供：保持现有无状态行为（向后兼容）。
- SSE 流中 `done` 事件额外携带 `conversation_id`，供前端回写 UI 状态。

**不改变 ReAct 流式逻辑**，只在 `done`/`error` 时做一次落库。

## 四、前端改动

### 4.1 auth hydrate 提前（修复工坊竞态根因）

**文件**：`frontend/src/main.tsx` 与 `frontend/src/stores/auth-store.ts`

- 在 `main.tsx` 的 `createRoot(...).render()` **之前**同步调用 `useAuthStore.getState().hydrate()`（及 `ui-store`、`workshop-store` 的 hydrate）。
- 移除 `App.tsx` 中 `useEffect` 里的 hydrate 调用（避免重复）。
- 效果：React 首次渲染时 `token` 已在 store 中，子组件 `useQuery` 触发时 `apiFetch` 能读到 token。

### 4.2 apiFetch 401 处理优化

**文件**：`frontend/src/lib/api.ts`

- 401 处理增加判断：仅当请求**确实带了 Authorization 头**（即 store 中原本有 token）时才调 `logout()`；若请求未带 token（说明未登录或 hydrate 未完成），不调 `logout()`，仅抛 `ApiError(401)`。
- 避免"从未登录"状态下破坏 localStorage。

### 4.3 useWorkshopSessions 加 enabled 守卫 + 登录后重取

**文件**：`frontend/src/hooks/use-workshop-sessions.ts`

```ts
const token = useAuthStore((s) => s.token);
return useQuery({
  queryKey: QK.sessions,
  queryFn: () => apiGet<WorkshopSessionSummaryDTO[]>("/api/workshop/sessions"),
  enabled: !!token,                 // 无 token 不查询
  staleTime: 30_000,
});
```

**文件**：`frontend/src/stores/auth-store.ts` 的 `setAuth` / `login` 成功后：
```ts
queryClient.invalidateQueries({ queryKey: ["workshop", "sessions"] });
queryClient.invalidateQueries({ queryKey: ["conversations"] });
```
（需在 store 中引入 `queryClient` 或通过事件通知）

## 五、后端代码架构优化

### 5.1 persistence.py 规范

`src/conversation_sessions/persistence.py` 参照 `workshop_sessions/persistence.py` 结构：
- 模块级 `Session = sessionmaker(bind=engine)`（复用 `src/db/database.py` 的 engine）。
- 每个 CRUD 函数独立 session，`try/finally` 关闭。
- `get_conversations_by_user(user_id, page, page_size)` 返回 `(items, total)`。
- `append_message(conversation_id, role, type, content, meta)` 事务内更新 `message_count` + `updated_at`。

### 5.2 SQLite 外键开启

**文件**：`src/db/database.py`

在 engine 创建后通过 `event.listens_for` 开启 `PRAGMA foreign_keys=ON`：
```python
from sqlalchemy import event
@event.listens_for(engine, "connect")
def _fk_on(dbapi_conn, _):
    cur = dbapi_conn.cursor()
    cur.execute("PRAGMA foreign_keys=ON")
    cur.close()
```
确保 `ON DELETE CASCADE` 生效。

### 5.3 路由层职责分离

`router.py` 仅做参数校验 + 调用 persistence + 返回 schema；业务逻辑（如标题默认值、迭代数统计）放 persistence 层。
